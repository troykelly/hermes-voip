# ADR-0013: SDES-SRTP media encryption (RFC 3711)

- **Date:** 2026-06-15
- **Status:** Accepted
- **Deciders:** agent session (W12 media-crypto)

## Context

ADR-0005 deferred the question of which media-security profile the plugin would implement,
pending the de-risking spike's capability matrix.  The result: the test gateway negotiates
**SDES-SRTP** (RFC 4568 `a=crypto` lines in TLS-protected SDP) and expects an
**RTP/SAVP** stream encrypted with the AES Counter Mode + HMAC-SHA1 SRTP transform
specified in **RFC 3711**.  ADR-0003 already parses and validates `a=crypto` attributes
(`CryptoAttribute`, `sdp.py`) and delivers the 30-octet base64-decoded master key‖salt
and the suite token (`AES_CM_128_HMAC_SHA1_80` or `AES_CM_128_HMAC_SHA1_32`) to the
caller.  `rtp.py` packs and parses plaintext RTP; it explicitly leaves SRTP to the
transport layer.

The implementation must meet several hard constraints:

- **No hand-rolled cipher.** AES must come from an audited library; writing our own AES
  implementation is a cryptographic anti-pattern banned by rule 19 (correctness) and
  security best practice.  `pyca/cryptography` (Apache-2.0) is the standard, audited Python
  crypto library and the correct vehicle for AES in Counter Mode.
- **Lean default install.**  `cryptography` is a heavy native dependency (OpenSSL binding)
  with its own transitive tree.  Pulling it into the default install would bloat the
  licence gate, the audit surface, and CI for every consumer that does not yet have a
  live media path.  The pattern already established by the `ml` extra is followed: a new
  `media` extra declares `cryptography`; the default install and base CI job remain lean.
- **No top-level import.** The existing pattern for optional extras uses `importlib` lazy
  loading so that `import hermes_voip.media.srtp` does not fail when the extra is absent
  and mypy stays clean in the no-extra environment.
- **Security-critical correctness.** RFC 3711 publishes known-answer test vectors (KAT) in
  Appendix B.  The implementation is only trusted after it reproduces those vectors
  exactly; any deviation at the KAT level means the cipher or key derivation is wrong,
  not merely functionally broken.
- **Key material never leaks.** Consistent with the `repr=False` discipline already applied
  to `CryptoAttribute.key_params` in `sdp.py`: key material must not appear in
  `repr`, log messages, exception text, or any string interpolation path.

## Decision

**Implement RFC 3711 SRTP** for suites `AES_CM_128_HMAC_SHA1_80` and
`AES_CM_128_HMAC_SHA1_32` in `src/hermes_voip/media/srtp.py` using:

- **`pyca/cryptography`** (`cryptography>=44.0.0,<46`) for AES in Counter Mode
  (`cryptography.hazmat.primitives.ciphers` — `AES` + `CTR` mode with a 128-bit key).
  AES-CM is not a primitive the library exposes directly; it is constructed as AES in CTR
  mode with the RFC 3711 §4.1.1 IV formula.  Licence: Apache-2.0 (dual Apache-2.0/BSD
  depending on OpenSSL binding choice; the `cryptography` wheel is Apache-2.0).
- **`hmac` + `hashlib` (stdlib)** for HMAC-SHA1 authentication tags (80-bit = 10 bytes,
  or 32-bit = 4 bytes, per suite).

`cryptography` is added to a **new `media` optional extra** in `pyproject.toml`, mirroring
the `ml` extra's isolation pattern.  The `import` inside `srtp.py` is deferred via
`importlib.import_module` so that `import hermes_voip.media.srtp` succeeds without the
extra (returning a clean module-level `ModuleNotFoundError` only when a session is
_constructed_).

### Key derivation (RFC 3711 §4.3, key_derivation_rate = 0)

From the 30-octet master key‖salt (`key[0:16]` + `salt[0:14]`), derive three per-session
keys using AES-CM with the label-based IV:

```
AES-CM( master_key, IV ) with IV = (label || 0x000000000000) XOR (0 || master_salt)
```

Labels (RFC 3711 Table 4):
- `0x00` → session encryption key (`k_e`), 128 bits
- `0x01` → session salt (`k_s`), 112 bits
- `0x02` → session authentication key (`k_a`), 160 bits

AES-CM for key derivation: encrypt 16-byte zero blocks at offset 0, 1, 2, … in counter
mode with the label-keyed IV to generate the required output length.

### Packet encryption (RFC 3711 §4.1)

AES Counter Mode IV for packet *n* (SSRC, ROC, SEQ):

```
IV = (k_s || 0x0000) XOR (0x0000 || SSRC || ROC || SEQ || 0x0000)
```
with `k_s` zero-padded to 128 bits (left 16 bits are 0x0000, then 14-byte salt).

The keystream is generated by AES-CTR with an internal counter starting at 0 for each
packet; the payload XOR keystream prefix encrypts/decrypts the payload.

**Only the RTP payload is encrypted.** The full clear RTP header — the 12-byte fixed
header plus any CSRC list (per `CC`) and any header extension (per the `X` bit) — stays in
the clear and is covered by the authentication tag (RFC 3711 §3.1/§3.3). The header length
is computed from the actual `CC`/`X` fields of each packet (`_rtp_header_len`), never
assumed to be 12 bytes; a fixed-12 assumption would corrupt any packet carrying CSRCs or
an extension. `protect(packet)` is sugar for `protect_wire(packet.pack())`; `protect_wire`
accepts a complete RTP datagram (so a relayed packet with CSRC/extension is handled).

### Authentication (RFC 3711 §4.2)

`protect` (via `protect_wire`):
1. Bind/verify the SSRC (see below), then encrypt the payload → ciphertext.
2. Authenticate: `HMAC-SHA1(k_a, clear_RTP_header ‖ ciphertext ‖ ROC_4bytes)`.
3. Append the most-significant `tag_len` bytes (10 for _80, 4 for _32).

`unprotect` (RFC 3711 §3.3 processing order):
1. Estimate the packet index (ROC) from the current receiver state.
2. Verify the auth tag FIRST (constant-time compare via `hmac.compare_digest`). The auth
   key derives from the master key alone — it is SSRC-independent — so a forged SSRC cannot
   forge a tag.
3. Enforce the per-SSRC binding (after auth, before any state mutation).
4. Replay-window check (64-packet sliding window, RFC 3711 §3.3).
5. Decrypt ciphertext → plaintext payload.
6. Return the `RtpPacket` reconstructed from the clear header + decrypted payload.

No receiver state (ROC, replay window, or SSRC binding) is mutated until authentication has
passed, so a forged first packet cannot poison the context.

### Per-SSRC cryptographic context (RFC 3711 §3.2.3)

The ROC and replay window form a **per-SSRC** context, so each `SrtpSession` serves a
single SSRC. The SSRC is fixed at construction (`ssrc=`) or captured from the first
authentic packet; a later packet whose SSRC differs is rejected (`SrtpError`) before any
state is mutated. For telephony there is one SSRC per stream direction, so this is the
normal case; a second media source needs its own session.

### SDES session parameters (lifetime / MKI)

RFC 4568 permits an inline key to carry `|lifetime` and/or `|MKI:length` after the base64
key‖salt. We implement neither, and an **MKI changes the SRTP packet layout** (MKI octets
sit between the payload and the auth tag) which this transform does not parse. Both are
therefore **rejected at `SrtpSession` construction** with a structural `SrtpError` (the
offending token is never echoed — it accompanies the key), rather than silently dropped.

### Rollover counter

The 16-bit sequence number wraps at 65535 → 0.  `SrtpSession` tracks the current rollover
counter (`roc`) and increments it on wraparound (RFC 3711 §3.3.1 index computation).

### Module shape

```python
# src/hermes_voip/media/srtp.py
class SrtpError(ValueError): ...

class SrtpSession:
    def __init__(self, crypto: CryptoAttribute, *, ssrc: int | None = None) -> None: ...
    def protect(self, packet: RtpPacket) -> bytes: ...        # = protect_wire(packet.pack())
    def protect_wire(self, rtp_wire: bytes) -> bytes: ...      # full datagram (CSRC/ext ok)
    def unprotect(self, data: bytes) -> RtpPacket: ...         # raises SrtpError on auth/replay/SSRC
```

`SrtpSession` is per-direction **and per-SSRC** (one inbound, one outbound per call, each
bound to that stream's SSRC).

## Consequences

- **Easier:** the SDES-SRTP media path is now complete end-to-end — `sdp.py` negotiates
  the key, `srtp.py` transforms packets, `rtp.py` provides the plaintext shape.  No new
  infrastructure (in-process only).  RFC 3711 KAT vectors provide a deterministic
  correctness proof, not just a round-trip smoke test.
- **Harder / committed to maintain:** we own a thin but security-critical SRTP
  implementation; any regression in key derivation or the auth-tag path breaks media
  security silently.  The KAT suite is a permanent regression gate.  The `cryptography`
  dependency brings an OpenSSL binding and a native build step, kept isolated in the
  `media` extra.
- **Licence (rule 35):** `cryptography>=44.0.0,<46` is Apache-2.0.  No GPL or
  copyleft component is introduced.
- **Not in scope:** DTLS-SRTP (RFC 5763/5764), ICE, SRTCP (control channel; deferred until
  a gateway requires it), Opus payload encryption, or any suite outside
  `AES_CM_128_HMAC_SHA1_{80,32}`.  SDES **MKI and non-default key lifetimes** are not
  supported — but, unlike the above, they are **actively rejected** at session construction
  (a silently-dropped MKI would desynchronise the packet layout), not merely absent.

## Alternatives considered

| Alternative | Rejected because |
| ----------- | ---------------- |
| Hand-rolled AES | Explicitly banned: writing your own AES is a known bad practice and violates AGENTS.md rule 19 (correctness proven by known-answer test) — any hand-rolled cipher will fail KAT unless it copies the RFC reference code, at which point the copy itself is the risk. |
| `pycryptodome` | Apache-2.0 / public-domain, but `pyca/cryptography` is more widely audited, is the standard for new Python crypto work, and is already present transitively in many Python environments. Using two crypto backends for no reason adds surface. |
| `aiortc`'s SRTP | `aiortc` keys SRTP via DTLS-SRTP (WebRTC profile); it does not expose SDES-SRTP key material externally. Pulling it in for SDES-SRTP would be the wrong abstraction level and add a large, WebRTC-specific dependency. |
| SRTCP | The test gateway does not require SRTCP; deferred until a gateway triggers that requirement. The KAT framework is extensible. |
| `media` extra in default install | Bloats licence gate + pip-audit for all consumers without a live media path. The `ml` extra precedent shows this is the correct isolation strategy. |
